- The 3 E2E tests in `runtime/bof/example_bofs_windows_test.go`
  (`TestExecute_HelloBeacon_E2E` / `_ParseArgs_E2E` / `_LoadLib_E2E`)
  are gated behind `testutil.RequireIntrusive(t)` and currently
  CRASH the test binary on the win10 TOOLS snapshot with
  `STATUS_ACCESS_VIOLATION` when MALDEV_INTRUSIVE=1. **Root cause
  diagnosed Linux-side; fix is well-scoped:**

  The BOF emits `mov rdi, [rip+disp32]` for `__imp_BeaconPrintf`
  references — a `REL32` (32-bit signed RIP-relative) addressing
  mode. Verified via `x86_64-w64-mingw32-objdump -d
  hello_beacon.o`:
  ```
  7:  48 8b 3d 00 00 00 00   mov  0x0(%rip),%rdi
                  a: IMAGE_REL_AMD64_REL32  __imp_BeaconPrintf
  ```

  **Status (commit `17a7399` + follow-ups)**: import-table
  co-allocation IS implemented. Multi-section loader IS
  implemented. REL32 addend handling IS implemented. Bare
  `__imp_<func>` (mingw default — no DLL prefix) resolver IS
  implemented (search order: kernel32 → advapi32 → user32
  → ws2_32 → ole32 → shell32).

  Current E2E status (MALDEV_INTRUSIVE=1):

    TestExecute_HelloBeacon_E2E  ✅ PASS
    TestExecute_LoadLib_E2E       ✅ PASS
    TestExecute_RealBOF_Nop       ✅ PASS
    TestExecute_ParseArgs_E2E     ✅ PASS (closed Windows-side
                                  2026-05-04 — see resolution
                                  below)

  **Resolved (2026-05-04, Windows pickup):** root cause was an
  endianness mismatch in the producer/consumer wire format —
  `Args.AddInt` / `AddShort` / `AddString` / `AddBytes` packed
  big-endian length prefixes while `BeaconDataInt` /
  `BeaconDataExtract` read them little-endian (matching the CS
  canonical `memcpy(&len, buf, 4)` native-int load used by
  TrustedSec COFFLoader / Outflank). The 4-byte string-length
  `0x0000000B` written BE was read LE as `0x0B000000` (184 MiB),
  tripping the overrun guard in `BeaconDataExtract` and
  returning NULL — the BOF then took the `if (s != 0)` false
  branch and never echoed the payload. Fix flipped both
  packers and unit-test assertions to little-endian.

  Historical pickup notes (no longer needed but kept for
  posterity):

  1. The BeaconDataExtract Go callback may be writing the
     out-length via a wrong-shape dereference. Confirm the
     `*(*int32)(unsafe.Pointer(outLenPtr)) = chunkLen` write
     reaches the BOF's `int s_len = 0` local — easiest path
     is a printf of `&s_len`'s address from the BOF, captured
     via BeaconPrintf, then a comparison post-Execute.
  2. The arg buffer may have moved between `Execute`'s
     `args[0]` pin and the BOF's read. Pin `args` for the
     duration of `syscallN` (currently the slice header is
     stack-local; CGO would `cgo.Handle` it).
  3. `BeaconDataExtract`'s 2-arg signature in `syscall.NewCallback`
     may interact poorly with Win64 ABI — the BOF could be
     passing the args via different register slots than the
     thunk expects.

  Recommend pickup: instrument the BOF directly via
  `BeaconPrintf("dbg: s_len=%d\\n", s_len)` (verbatim format,
  no expansion — but the literal helps confirm the BOF reaches
  the if-branch), build with mingw, run via
  `MALDEV_INTRUSIVE=1 go test -count=1 -v
  -run TestExecute_ParseArgs ./runtime/bof`.
