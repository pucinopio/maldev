## Recommended next moves

In priority order:

1. **P2.6 SpoofCallHWBP** — biggest backlog row left. Fortra-style HW-breakpoint callstack variant. ~400-600 LOC + tests. Risk: existing `SpoofCall` is EXPERIMENTAL with a documented crash path; the HWBP variant adds VEH + DR0-3 + single-stepping complexity. Author with `MALDEV_SPOOFCALLHWBP_E2E=1` gating until verified on a real target.
2. **Upgrade `cert.Forge` to produce real Authenticode** — now that `(*Certificate).Parse()` shipped (v0.42.0), the missing piece is computing the PE's Authenticode hash + wrapping it in a Microsoft-OID `SpcIndirectDataContent`. saferwall.File.Authentihash() already gives the hash; need to ASN.1-encode the SpcIndirectDataContent and feed it as `Forge.opts.Content`. ~100 LOC. Operators with stolen leaf keys then get signatures that pass `signtool verify`.
3. **P2.12 (3 rows) — masquerade preset binaries** — extend `pe/masquerade/internal/gen` to accept non-System32 paths, then add presets for msedge, AcroRd32, WindowsTerminal, OneDrive, Office. Repetitive but ships fast.
4. **`(*pe/parse.File).Overlay()` exposure** — saferwall surfaces `OverlayOffset` + `Overlay()` + `NewOverlayReader()`. Useful for "binary has appended payload" detection (common in droppers). ~20 LOC + 1 test.
5. **Recon-side `cert.Read(pePath).Parse()` ergonomic chain** — the current API forces a 2-step `Read` then `Parse`. A `cert.Inspect(pePath) (*ParsedAuthenticode, error)` wrapper would be a one-call operator surface. ~10 LOC.
