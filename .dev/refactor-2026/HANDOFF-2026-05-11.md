## Empirical operational envelope (verified by Win10 VM E2E)

| Payload class | Vanilla | `RandomizeAll` | Notes |
|---|---|---|---|
| Go static-PIE (clean exit) | ✅ | ✅ | `winhello.exe` |
| Go static-PIE (panic + recover) | ✅ | ✅ | `winpanic.exe` — Go uses pclntab, not Win32 SEH |
| mingw `-nostdlib` Win32 (no CRT) | ✅ | ✅ | `winhello_w32.exe` — proves IMPORT walker is enough |
| mingw default (with CRT) | ❌ at PackBinary | same | TLS callbacks → `transform.ErrTLSCallbacks` |
| DLLs (any toolchain) | ❌ at PackBinary | same | DllMain contract mismatch → `transform.ErrIsDLL` (v0.108.0) |
| MSVC CFG-protected (modern Microsoft) | ❌ at runtime | same | CFG cookie → `0xC0000409` |

**Workaround for the broken classes:** use
`PackBinaryBundle` + `cmd/bundle-launcher` instead of in-place
`PackBinary` — preserves the original `.text` intact, CFG and
TLS unaffected.

## Walker suite — final disposition

The 7-slice walker suite I scoped this morning (~1820 LOC,
v0.105.0–v0.111.0) is now classified as **on-demand only**.
Each future slice ships only when an operator hits a real
payload that fails for a specific directory's staleness.
Trigger conditions documented in
`.dev/refactor-2026/packer-2f3c-walker-suite-plan.md`:
- 2-F-3-c-3 EXCEPTION — only when C/C++ payload uses SEH
- 2-F-3-c-4 LOAD_CONFIG — only when payload has CFG (and even
  then, CFG fails vanilla too — likely no-fix)
- 2-F-3-c-5 EXPORT — only when packing DLLs
- 2-F-3-c-6 RESOURCE — only when payload calls `FindResource`
- 2-F-3-c-7 DEBUG — purely cosmetic for tools
- 2-F-3-c-8 DELAY_IMPORT — only for `/DELAYLOAD` payloads
