## Open questions for next session

1. **MSVC fixture for richer DLL E2E** — `BuildDLLWithReloc` is
   minimal (no exports). To validate `GetProcAddress` on
   user-defined exports through `FormatWindowsDLL` (not just
   forwarders) we'd need a `testlib.dll` built on the Win10 VM
   via `cl.exe`. Provisioning MSVC on the VM is the prereq.
2. **OPSEC cleanup pass** — the packed binary still carries some
   identifiable artifacts (the `.mldv` default stub section
   name, characteristics flags, etc.). `RandomizeAll` already
   covers the easy wins; a stride deeper would need to also
   randomise stub-internal layout (e.g. instruction reordering
   that's not register-scope).
3. **Phase 3 anti-static-RE cover** — junk content fill in BSS
   separators, fake imports to legit-looking DLLs, optional
   stub control-flow flattening. Composes with everything
   shipped today. Different threat model (anti-RE vs anti-
   fingerprint).
4. **`PackProxyDLL` for non-Go payloads** — strict E2E used
   `probe_converted.exe` (mingw `-nostdlib`). Should also
   validate against a Go static-PIE input (winhello.exe). Likely
   trivial since the converted-DLL path under the hood is the
   same.
5. **MSVC `cl.exe` / DOS stub variants** — the `dllproxy.Options.DOSStub`
   flag exists but isn't exercised in `PackProxyDLL` yet. A
   quick wire would let operators emit fused proxies that look
   like MSVC-linked DLLs at the DOS-stub byte level.
